\section{Diseño del sistema}

\subsection{Requisitos y decisiones de diseño del sistema}

En esta sección se definen los requisitos que debe cumplir el sistema de sonificación EEG--MIDI desarrollado en este Trabajo Fin de Grado. Estos requisitos permiten delimitar qué debe hacer el prototipo, bajo qué condiciones debe operar y qué decisiones de diseño se han adoptado en la implementación final.

Es importante diferenciar entre requisitos funcionales, requisitos no funcionales y decisiones de diseño. Los requisitos funcionales describen las capacidades que debe ofrecer el sistema; los requisitos no funcionales recogen propiedades de calidad, seguridad, modularidad y escalabilidad; y las decisiones de diseño explican cómo se ha implementado el sistema en esta versión, sin convertir dichas elecciones en limitaciones cerradas de la arquitectura.

El sistema se plantea como un prototipo experimental de sonificación EEG--MIDI en vivo. Su finalidad no es realizar diagnóstico médico ni evaluar patologías neurológicas, sino transformar señales EEG reales en eventos musicales enviados mediante una salida MIDI física.

\subsubsection{Requisitos funcionales}

Los requisitos funcionales definen las capacidades principales que debe ofrecer el sistema. En este proyecto abarcan la cadena completa de funcionamiento: adquisición EEG real, transmisión de muestras, procesamiento, extracción de características, generación musical y salida MIDI física.

\begin{description}[leftmargin=!,labelwidth=1.8cm]

    \item[RF-1] \textbf{Adquisición EEG en vivo.} El sistema debe permitir la adquisición de señales EEG reales procedentes de un usuario durante una sesión de funcionamiento. La señal utilizada para la sonificación debe proceder de una captura en vivo y no de la reproducción \textit{offline} de bases de datos o registros previamente almacenados.

    Este requisito responde al objetivo principal del proyecto: transformar actividad EEG real en eventos musicales MIDI durante el funcionamiento del prototipo.

    \item[RF-2] \textbf{Adquisición multicanal de señales EEG.} El sistema debe permitir la adquisición de señales EEG en varios canales de forma simultánea. Aunque la sonificación inicial se base en un canal principal, la adquisición multicanal permite conservar información adicional y facilita futuras ampliaciones.

    La implementación desarrollada utiliza el ADS1299--4PAG, pero el requisito se centra en la capacidad multicanal del sistema, no en cerrar la arquitectura a un único componente.

    \item[RF-3] \textbf{Configuración flexible de adquisición.} El sistema debe permitir configurar parámetros de adquisición, especialmente la frecuencia de muestreo, para adaptarse a distintas necesidades de análisis EEG.

    La frecuencia de \SI{250}{\hertz} utilizada en las pruebas del TFG se considera una decisión de diseño, no una limitación funcional del sistema.

    \item[RF-4] \textbf{Compatibilidad con distintos montajes EEG de superficie.} El sistema debe permitir la conexión de electrodos en diferentes configuraciones EEG de superficie, incluyendo posiciones compatibles con el sistema internacional 10--20.

    La configuración ear--EEG/mastoides con referencia en muñeca izquierda corresponde al montaje empleado en la validación, no a una restricción funcional obligatoria.

    \item[RF-5] \textbf{Transmisión continua de muestras.} El sistema debe transmitir de forma continua las muestras EEG digitalizadas desde el bloque de adquisición hacia el bloque de procesamiento.

    Este requisito garantiza que la señal capturada pueda analizarse durante la sesión activa y emplearse posteriormente en la generación musical.

    \item[RF-6] \textbf{Procesamiento continuo de la señal EEG.} El sistema debe procesar de forma continua las muestras EEG recibidas, organizándolas en buffers o ventanas temporales.

    La implementación actual se realiza en Python sobre Arduino UNO Q, pero el requisito funcional es el procesamiento continuo, no el lenguaje empleado.

    \item[RF-7] \textbf{Extracción de características EEG.} El sistema debe extraer características temporales y espectrales de la señal EEG adquirida, como amplitud, RMS, potencia absoluta, potencia relativa y frecuencias pico.

    Estas características actúan como representación intermedia entre la señal EEG cruda y el módulo de sonificación.

    \item[RF-8] \textbf{Análisis por bandas EEG.} El sistema debe obtener información asociada a bandas EEG de interés, como delta, theta, alfa, beta y gamma.

    Estas bandas permiten describir el contenido frecuencial de la señal y utilizarlo como entrada para el mapeo musical.

    \item[RF-9] \textbf{Generación de parámetros musicales a partir de características EEG.} El sistema debe transformar las características EEG extraídas en parámetros musicales como altura, duración, intensidad, escala, tonalidad o densidad rítmica.

    Este requisito constituye el núcleo de la sonificación EEG--MIDI, al convertir información cerebral en decisiones musicales.

    \item[RF-10] \textbf{Organización musical jerárquica.} El sistema debe organizar la generación musical mediante una estructura jerárquica formada por segmentos, compases y notas.

    Esta organización evita una conversión directa de señal a sonido y permite generar una salida musical más estructurada.

    \item[RF-11] \textbf{Generación de eventos MIDI válidos.} El sistema debe generar eventos MIDI válidos, especialmente mensajes equivalentes a \textit{Note On} y \textit{Note Off}.

    Cada nota debe tener inicio, duración y finalización definidos para evitar mensajes incompletos o notas colgadas.

    \item[RF-12] \textbf{Salida MIDI física.} El sistema debe enviar los eventos musicales generados mediante una salida MIDI física.

    En la implementación desarrollada se emplea la línea asociada al pin D1/RX y la PCB MIDI, pero el requisito principal es disponer de una salida física operativa.

    \item[RF-13] \textbf{Monitorización auxiliar del sistema.} El sistema debe disponer de una interfaz auxiliar de monitorización que permita observar el estado general del prototipo durante su funcionamiento.

    La interfaz web actúa como apoyo para supervisar adquisición, procesamiento y generación musical, sin sustituir a la salida MIDI física.

    \item[RF-14] \textbf{Separación entre adquisición, procesamiento y sonificación.} El sistema debe mantener una separación funcional entre adquisición, procesamiento de señal y sonificación.

    Esta separación facilita la escalabilidad del sistema y permite modificar un bloque sin reescribir completamente los demás.

\end{description}

\subsubsection{Requisitos no funcionales}

Los requisitos no funcionales describen las condiciones de calidad, operación y diseño que debe cumplir el sistema sin imponer una implementación concreta. En este proyecto son especialmente importantes porque el prototipo trabaja con señales EEG reales, de baja amplitud y sensibles al ruido, y debe transformarlas en eventos MIDI en vivo.

\begin{description}[leftmargin=!,labelwidth=2cm]

    \item[RNF-1] \textbf{Funcionamiento en tiempo real.} El sistema debe operar de forma continua durante una sesión de adquisición y sonificación, manteniendo una respuesta temporal suficiente para que los cambios en la señal EEG puedan reflejarse en la salida musical.

    Este requisito no impone una frecuencia de muestreo o una ventana de análisis concreta, sino que exige que la cadena completa funcione sin depender de un análisis \textit{offline} posterior.

    \item[RNF-2] \textbf{Seguridad y uso no clínico.} El sistema debe estar orientado a un uso experimental y no clínico. Su finalidad es la sonificación de señales EEG reales en vivo, no el diagnóstico médico ni la toma de decisiones clínicas.

    Al conectarse al cuerpo humano mediante electrodos, el prototipo debe emplear alimentación de baja tensión y medidas de diseño orientadas a reducir riesgos eléctricos en un entorno controlado y supervisado.

    \item[RNF-3] \textbf{Robustez frente al ruido e interferencias.} El sistema debe diseñarse teniendo en cuenta la baja amplitud de las señales EEG y su sensibilidad frente al ruido eléctrico, los artefactos fisiológicos y las interferencias externas.

    Este requisito afecta al diseño hardware, al montaje de adquisición y al procesamiento posterior de la señal.

    \item[RNF-4] \textbf{Modularidad del sistema.} El sistema debe organizarse en bloques funcionales separados: adquisición, comunicación, procesamiento, sonificación, salida MIDI e interfaz auxiliar.

    Esta separación permite modificar una parte del sistema sin afectar de forma directa al resto.

    \item[RNF-5] \textbf{Escalabilidad.} El sistema debe permitir futuras ampliaciones tanto a nivel hardware como software.

    Entre las posibles ampliaciones se incluyen el uso de más canales EEG, nuevos mapeos musicales, selección dinámica de canal, nuevas visualizaciones o adaptación a distintos dispositivos MIDI.

    \item[RNF-6] \textbf{Bajo coste y accesibilidad.} El sistema debe priorizar el uso de componentes y herramientas accesibles dentro de un contexto académico y de prototipado.

    El objetivo no es competir con sistemas comerciales de EEG, sino desarrollar un prototipo funcional, documentable y reproducible con recursos razonables.

    \item[RNF-7] \textbf{Replicabilidad y documentación.} El sistema debe estar suficientemente documentado para que su funcionamiento pueda ser entendido, reproducido y modificado posteriormente.

    La documentación debe cubrir la arquitectura general, conexiones hardware, configuración de adquisición, procesamiento de señal, generación MIDI y pruebas realizadas.

    \item[RNF-8] \textbf{Mantenibilidad del código.} El código del sistema debe organizarse de forma comprensible y mantenible, evitando dependencias innecesarias entre módulos.

    Esta propiedad facilita la depuración y la evolución del sistema durante ajustes posteriores.

    \item[RNF-9] \textbf{Portabilidad del prototipo.} El sistema debe poder funcionar como un prototipo compacto y transportable, adecuado para realizar pruebas en un entorno experimental sin equipamiento de laboratorio complejo.

    La alimentación mediante USB y la integración en Arduino UNO Q favorecen esta portabilidad.

\end{description}

\subsubsection{Decisiones de diseño adoptadas}

Una vez definidos los requisitos funcionales y no funcionales, se recogen las principales decisiones de diseño adoptadas durante el desarrollo. Estas decisiones representan la forma concreta en la que se ha implementado el sistema en este TFG, pero no deben confundirse con requisitos cerrados, ya que el objetivo es mantener una arquitectura escalable.

\begin{description}[leftmargin=!,labelwidth=1.8cm]

    \item[DD-1] \textbf{Uso del ADS1299--4PAG como front-end de adquisición.} Aunque el requisito funcional es disponer de adquisición multicanal de señales EEG, en este trabajo se ha seleccionado el ADS1299--4PAG como circuito principal de adquisición.

    Esta elección se debe a que está orientado a la medida de biopotenciales y permite digitalizar señales EEG de baja amplitud con varios canales disponibles.

    \item[DD-2] \textbf{Frecuencia de muestreo de 250 Hz en las pruebas.} La librería desarrollada para la comunicación con el ADS1299 permite configurar diferentes frecuencias de muestreo. Sin embargo, para las capturas y resultados de este TFG se ha utilizado una frecuencia de \SI{250}{\hertz} por canal.

    Esta configuración resulta suficiente para el análisis de las bandas EEG empleadas en la sonificación y mantiene una carga de procesamiento razonable.

    \item[DD-3] \textbf{Configuración experimental ear--EEG/mastoides.} El sistema permite la conexión de electrodos en diferentes configuraciones EEG de superficie. No obstante, durante la validación se ha utilizado una configuración próxima a ear--EEG/mastoides, con referencia o BIAS en la muñeca izquierda.

    Esta configuración se seleccionó por su estabilidad práctica, facilidad de colocación y menor interferencia con el cabello.

    \item[DD-4] \textbf{Procesamiento en Python dentro de Arduino UNO Q.} El requisito funcional es procesar la señal EEG de forma continua y extraer características útiles para la sonificación. En la implementación desarrollada, este procesamiento se realiza en Python dentro del entorno Linux/Qualcomm de Arduino UNO Q.

    Esta decisión facilita el desarrollo, la depuración y la modificación de los algoritmos de análisis y sonificación.

    \item[DD-5] \textbf{Estimación espectral mediante multitaper.} El sistema requiere obtener características espectrales robustas a partir de la señal EEG. Para ello, en esta versión se ha empleado el método multitaper como estrategia principal de estimación espectral.

    Este método permite reducir la varianza de la estimación y los efectos asociados al ventaneado en señales EEG no estacionarias.

    \item[DD-6] \textbf{Ventana de análisis de aproximadamente 4 segundos.} Durante el desarrollo se ha utilizado una ventana temporal de análisis de aproximadamente \SI{4}{\second}.

    Esta configuración busca un equilibrio entre estabilidad frecuencial y capacidad de respuesta temporal durante la sonificación.

    \item[DD-7] \textbf{Salida MIDI física mediante pin D1/RX y PCB MIDI.} El requisito funcional es disponer de salida MIDI física. En la implementación desarrollada, los eventos MIDI generados se envían mediante la línea asociada al pin D1/RX hacia la PCB MIDI del prototipo.

    Esta decisión permite conectar el sistema con dispositivos musicales externos compatibles con MIDI.

    \item[DD-8] \textbf{Interfaz web como herramienta auxiliar.} La interfaz web se ha planteado como una herramienta auxiliar de monitorización, no como la salida principal del sistema.

    Su función es facilitar la supervisión del estado del prototipo durante las pruebas.

    \item[DD-9] \textbf{Matriz LED como complemento visual opcional.} La matriz LED se considera un complemento visual opcional para representar de forma sencilla la actividad musical generada.

    No constituye un requisito funcional esencial para cumplir el objetivo principal del sistema.

\end{description}
